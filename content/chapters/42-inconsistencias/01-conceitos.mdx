Inconsistências de dados são um dos problemas mais insidiosos em sistemas bancários. Podem surgir de falhas técnicas, erros humanos, ou condições de corrida em sistemas distribuídos. Detectá-las e corrigi-las é uma das funções mais críticas do backoffice.

## O que é uma inconsistência

Uma inconsistência ocorre quando dois sistemas (ou duas partes do mesmo sistema) têm visões diferentes da mesma realidade.

**Exemplos:**
- O sistema de contas diz que o saldo é R$ 1.000, mas o ledger tem lançamentos que somam R$ 1.100
- O sistema de crédito diz que o contrato está ativo, mas o COSIF não tem a provisão correspondente
- O banco registrou um Pix como processado, mas o SPI não tem registro da transação
- O cliente A transferiu R$ 500 para o cliente B, mas o débito foi feito mas o crédito não

## Causas de inconsistências

### Falhas em transações distribuídas

Em sistemas distribuídos, uma operação pode envolver múltiplos sistemas. Se um sistema é atualizado mas outro não (por falha de rede, timeout, ou crash), surge uma inconsistência.

**Exemplo:** um Pix debita a conta do pagador (sistema de contas) mas falha antes de gerar o lançamento contábil (ledger). O saldo está correto, mas o registro contábil está incompleto.

### Condições de corrida (race conditions)

Duas operações simultâneas que afetam os mesmos dados podem interferir uma na outra.

**Exemplo:** dois saques simultâneos de R$ 600 em uma conta com saldo de R$ 1.000. Ambos passam na validação de saldo (R$ 1.000 > R$ 600), mas juntos excedem o saldo. O resultado é saldo negativo.

### Erros de arredondamento

Operações com casas decimais podem gerar pequenas diferenças de arredondamento que se acumulam ao longo do tempo.

**Exemplo:** juros calculados com 4 casas decimais no sistema de crédito, mas registrados com 2 casas no ledger. A diferença de centavos se acumula em milhares de contratos.

### Reprocessamento sem idempotência

Uma operação é reprocessada (por retry após falha) sem verificar se já foi processada antes. O resultado é um lançamento duplicado.

### Migração de dados

Durante migrações de sistema, dados podem ser transformados incorretamente ou perdidos.

## Detecção de inconsistências

### Reconciliações automáticas

O sistema compara automaticamente os saldos entre sistemas:
- Saldo no sistema de contas vs. soma dos lançamentos no ledger
- Saldo no ledger vs. saldo no COSIF
- Posição interna vs. posição no BACEN (STR, SPI)

Divergências são alertadas automaticamente.

### Verificações de integridade

Verificações periódicas que garantem que os dados estão internamente consistentes:
- Todos os contratos de crédito têm provisão correspondente?
- Todos os Pix confirmados têm lançamento contábil?
- Todos os clientes ativos têm conta ativa?

### Análise de anomalias

Detecção de padrões anômalos que podem indicar inconsistências:
- Contas com saldo negativo (sem cheque especial)
- Contratos com saldo devedor negativo
- Lançamentos com valor zero

## Correção de inconsistências

A correção depende do tipo e da gravidade da inconsistência:

**Lançamento faltando:** criar o lançamento com data retroativa, documentando o motivo.

**Lançamento duplicado:** estornar o lançamento duplicado.

**Saldo incorreto:** ajuste manual com dupla aprovação e documentação.

**Inconsistência sistêmica:** pode exigir um processo de correção em lote para todos os registros afetados.

**Princípio fundamental:** a correção deve ser feita de forma auditável. Nunca modifique registros existentes — sempre crie novos registros de correção.

## Prevenção de inconsistências

A melhor correção é a prevenção:

- **Idempotência:** garantir que operações podem ser reprocessadas sem efeitos colaterais
- **Transações atômicas:** usar transações de banco de dados para garantir que múltiplas atualizações são feitas juntas ou não são feitas
- **Padrão Outbox:** garantir que eventos são publicados na mesma transação que a operação — evita o problema de "operação gravada mas evento não publicado"
- **Padrão Saga com compensação:** em fluxos distribuídos, definir transações compensatórias para cada etapa, garantindo que falhas parciais sejam revertidas de forma consistente
- **Reconciliações frequentes:** detectar inconsistências rapidamente, antes que se acumulem
- **Testes de carga:** verificar que o sistema se comporta corretamente sob alta concorrência
