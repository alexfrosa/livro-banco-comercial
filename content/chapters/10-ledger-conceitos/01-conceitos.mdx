O ledger (livro-razão) é o coração contábil de um banco. É onde cada centavo movimentado é registrado de forma permanente e auditável. Entender os princípios do ledger é fundamental para qualquer sistema financeiro.

## Partidas dobradas (double-entry)

O sistema de partidas dobradas tem mais de 500 anos — foi formalizado por Luca Pacioli em 1494 e ainda é a base de toda a contabilidade moderna.

O princípio é simples: **cada operação financeira afeta pelo menos duas contas**, com o total de débitos sempre igual ao total de créditos.

**Exemplo: depósito de R$ 1.000 em conta corrente**

```
Débito:  Caixa (ativo)                    R$ 1.000
Crédito: Depósitos à Vista (passivo)      R$ 1.000
```

O banco recebeu R$ 1.000 em caixa (ativo aumentou) e deve R$ 1.000 ao cliente (passivo aumentou). O balanço se mantém equilibrado.

**Exemplo: empréstimo de R$ 5.000**

```
Débito:  Operações de Crédito (ativo)     R$ 5.000
Crédito: Conta Corrente do Cliente (passivo) R$ 5.000
```

O banco tem um crédito a receber (ativo aumentou) e creditou na conta do cliente (passivo aumentou).

### Por que partidas dobradas?

O sistema de partidas dobradas tem uma propriedade matemática poderosa: **a soma de todos os débitos deve sempre ser igual à soma de todos os créditos**. Se não for, há um erro em algum lugar.

Isso torna o sistema auto-verificável. Qualquer inconsistência — um lançamento faltando, um valor errado — quebra o equilíbrio e pode ser detectada.

## Imutabilidade dos registros

No ledger bancário, **registros não são modificados — são corrigidos com novos lançamentos**.

Se um lançamento foi feito incorretamente, a correção é feita com um lançamento de estorno (que reverte o lançamento original) seguido de um novo lançamento correto. O lançamento original permanece no histórico.

**Por que imutabilidade?**

- **Auditoria:** é possível reconstruir o histórico completo de qualquer conta em qualquer ponto no tempo
- **Regulação:** o BACEN exige que os registros contábeis sejam mantidos e não alterados
- **Integridade:** se registros pudessem ser modificados, seria impossível detectar fraudes ou erros

**Implementação técnica:**

Em bancos de dados relacionais, isso significa que a tabela de lançamentos tem apenas operações de INSERT — nunca UPDATE ou DELETE. Cada lançamento tem um timestamp imutável e um ID único.

```sql
-- Correto: INSERT para cada lançamento
INSERT INTO lancamentos (id, conta, tipo, valor, timestamp, descricao)
VALUES ('abc-123', '12345', 'DEBITO', 500.00, NOW(), 'Saque ATM');

-- Errado: UPDATE modifica o histórico
UPDATE lancamentos SET valor = 600.00 WHERE id = 'abc-123'; -- NUNCA FAÇA ISSO
```

## Idempotência no ledger

Idempotência significa que executar a mesma operação múltiplas vezes produz o mesmo resultado que executá-la uma única vez.

No contexto do ledger, isso é crítico para lidar com falhas de rede e retentativas. Se um sistema envia a instrução de um lançamento e não recebe confirmação (timeout), ele precisa poder reenviar sem risco de duplicar o lançamento.

**Implementação:**

Cada instrução de lançamento inclui um **ID de idempotência** (também chamado de chave de idempotência ou correlation ID). O ledger verifica se já processou um lançamento com aquele ID antes de processar novamente.

```json
{
  "idempotency_key": "pix-abc-123-2024-01-15",
  "conta_debito": "12345",
  "conta_credito": "67890",
  "valor": 500.00,
  "descricao": "Pix para João"
}
```

Se essa instrução chegar duas vezes, o ledger processa apenas uma vez e retorna o mesmo resultado para ambas as chamadas.

## Consistência eventual vs. consistência forte

Em sistemas distribuídos, há uma tensão entre performance e consistência:

**Consistência forte:** toda leitura reflete a escrita mais recente. O saldo que você vê é sempre o saldo real. Mais seguro, mas mais lento (requer coordenação entre nós do sistema).

**Consistência eventual:** as escritas se propagam gradualmente. Por um breve momento, diferentes partes do sistema podem ver valores diferentes. Mais rápido, mas requer cuidado para evitar problemas.

Para operações críticas como débitos (saques, pagamentos), os bancos geralmente usam consistência forte — não pode haver dúvida sobre o saldo disponível. Para operações menos críticas (atualização de extrato, notificações), consistência eventual é aceitável.
